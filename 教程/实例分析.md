# Leash 实例分析（逐行走读）

> 前面各章讲语法、讲标准库，都是"自上而下"地教。本章反过来：挑两个**真实可运行**
> 的程序，一行一行地拆开看它们是怎么用 Leash 把想法变成代码的。读完这一章，你应该
> 能照猫画虎写出自己的第一个实用脚本。
>
> 所有代码都在仓库里：`examples/todo.ae`（单文件极简示例）和 `examples/agent/`
> （多文件 Agent 示例）。建议边读边跑。

---

## 0. 本章看什么

| 实例 | 文件 | 量级 | 重点 |
|------|------|------|------|
| 一、待办清单 | `examples/todo.ae` | 单文件 ~60 行 | 变量 / 函数 / list / map / file / 控制流 |
| 二、多文件 Agent | `examples/agent/*` | 6 文件 | 多文件 import / 原生包 / 全局配置注入 / 交互 vs 演示 |

实例一刻意做得"土味"一点：没有花哨语法，全是日常脚本最常用的那几样。把这一关过
了，实例二的多文件工程化写法就只是"同一套积木拼得更大"而已。

## 1. 实例一：待办清单 `todo.ae`

先上完整代码（已实测可运行）：

````leash
// todo.ae — 极简待办清单（单文件示例）
package file
import "list"

fn load(path: str) -> list
  if exists(path)
    return parse(read(path))   // read 返回字符串，parse 解析成 list/map
  return []

fn save(path: str, items: list)
  write(path, stringify(items))

fn show(items: list)
  out("待办清单 (共 " + len(items) + " 项):")
  for i, it in items
    let mark = "  "
    if it["done"]
      mark = "[x]"
    let text = it["text"]
    out("  " + (i + 1) + ". " + mark + " " + text)

fn main
  let path = "todo.json"
  let items = load(path)
  while true
    out("")
    out("命令: a 添加 | l 列表 | d 完成 | q 退出")
    out("> ")
    let cmd = in()
    if cmd == "q"
      break
    else if cmd == "a"
      out("任务内容: ")
      let t = in()
      items = list_push(items, {"text": t, "done": false})
      save(path, items)
      out("已添加。")
    else if cmd == "l"
      show(items)
    else if cmd == "d"
      out("完成第几项? ")
      let n = parse(in())
      if n >= 1 and n <= len(items)
        let new_items = []
        for i, it in items
          if i + 1 == n
            new_items = list_push(new_items, {"text": it["text"], "done": true})
          else
            new_items = list_push(new_items, it)
        items = new_items
        save(path, items)
        out("已标记完成。")
      else
        out("序号无效。")
    else
      out("未知命令。")
  out("已保存到 " + path + "，再见。")
````

跑起来：

````bash
./leash examples/todo.ae
# 然后依次输入: a 回车 写教程 回车  a 回车 跑步 回车  l 回车  d 回车 1 回车  q 回车
````

### 1.1 第一行：声明需要的能力

````leash
package file
import "list"
````

Leash 的设计哲学是"**默认什么都不能做**"。你的程序要碰文件，就得显式声明 `package file`；
要用到 `list_push` 这类标准库函数，就得 `import "list"`。这两行写在第一行附近，相当于
"向宿主申请能力"。如果漏写 `package file` 却在 `fn main` 里调用了 `read`，编译器会直接报
"函数 … 缺少能力: io"——这正是 Leash 的"能力门控"在起作用。

> 注意区别：`package file` 引入的是**原生包**（由 C++ 实现、全局可用的裸名函数如
> `read`/`write`/`exists`）；`import "list"` 引入的是**纯 Leash 标准库包**（`lib/list.ae`），
> 它导出的 `list_push` 等也是裸名，但实现是 Leash 代码。调用时一律写裸名，不写前缀。

### 1.2 数据模型：用 list 装 map

任务清单本质是一个列表，每个任务是一个带两个字段的小对象：

````leash
items = list_push(items, {"text": t, "done": false})
````

- `{"text": t, "done": false}` 是一个**映射字面量**（map），键是字符串、值可以是任意
  表达式（`t` 是变量）。
- 列表**没有非空字面量**语法（也就是不能写 `[a, b, c]`），统一用 `list_push` 往里加。
  而且 `list_push` **返回新的列表**，所以要写成 `items = list_push(items, ...)` 把结果接
  回来——这是 Leash 里 list/map 多为"值语义"的体现，别写成没有返回值的语句。

为什么"完成一项"要重建整个列表？看这段代码：

````leash
let new_items = []
for i, it in items
  if i + 1 == n
    new_items = list_push(new_items, {"text": it["text"], "done": true})
  else
    new_items = list_push(new_items, it)
items = new_items
````

原因是 Leash 当前**不支持链式下标赋值**（即 `items[n]["done"] = true` 这种写法解析不了）。
于是我们用"读出来、改好、再放回去"的方式：遍历原列表，遇到要改的那一项就生成一份带
`"done": true` 的新 map，其余原样搬运，最后整体替换。写法啰嗦一点，但完全可运行，也
更贴近"数据不可变"的函数式思路。

### 1.3 map 的读写

````leash
let text = it["text"]     // 读字段
if it["done"]             // 读布尔字段做判断
````

读 map 用 `m["键"]`。键必须是字符串字面量或简单变量。把结果先存进 `let text` 再拼字符串，
比直接把 `it["text"]` 塞进插值 `{}` 更稳——复杂表达式进插值偶尔会踩解析的边角，养成"先算
出来再拼"的习惯能少踩很多坑。

### 1.4 控制流：while + if/else if 主循环

````leash
while true
  ...
  let cmd = in()
  if cmd == "q"
    break
  else if cmd == "a"
    ...
  else if cmd == "l"
    show(items)
  ...
````

- `while true` 是 Leash 里写"交互主循环"的常规套路（没有 `loop` 关键字）。
- `in()` 读一行用户输入，**只能在 `fn main` 里用**（它属于宿主注入的 io 能力）。
- `else if` 连续下探实现"菜单分发"，注意是 `else if` 而不是 `elif`。
- `break` 退出循环；`continue` 也能用，本例没用到。

### 1.5 字符串拼接而非插值

````leash
out("  " + (i + 1) + ". " + mark + " " + text)
````

这里故意用 `+` 拼接而不是 `"{i+1}. {mark} {text}"` 插值。两者都合法，但**字符串和整数
不能直接比较**，拼接时整数会被转成字符串；插值 `{}` 里也能放表达式。本项目里两种都常用，
选顺手的即可。一个硬性坑：**字符串之间不能做 `<` `>` 关系比较**（`c >= "0"` 会报错），要
比较字符大小得用 `ord()` 转成整数。

### 1.6 持久化：file 包 + JSON

````leash
fn load(path: str) -> list
  if exists(path)
    return parse(read(path))
  return []

fn save(path: str, items: list)
  write(path, stringify(items))
````

- `exists` / `read` / `write` 都是 `file` 包的裸名函数（注意 `read(path)` 只接收一个
  参数，没有"默认值"第二参）。
- `stringify` 把 list/map 序列化成 JSON 字符串，`parse` 反向解析。它们来自 `core`/`json`
  包（自动注入，无需 import）。
- 文件读写路径**相对进程当前工作目录（cwd）**。你在仓库根跑 `./leash examples/todo.ae`，
  那 `todo.json` 就落在仓库根；从 leash-studio 运行则落在文件所在目录。运行完记得清理。

**小结**：这一个 ~60 行的小程序，已经覆盖了"写一个实用脚本"所需的全部基础构件——能力
声明、函数、list、map、循环、分支、字符串、文件持久化。把它读熟，你就过了 Leash 的第一关。

## 2. 实例二：多文件 Agent 示例 `examples/agent/`

实例一的全部代码塞在一个文件里。一旦程序变大，单文件就会又长又难维护。Leash 用
**多文件 `import`** 把代码拆成多个 `.ae`，本例就是这么组织的。先看清目录：

````
examples/agent/
├── main.ae      入口：TUI 主循环、交互/演示分支
├── config.ae    配置加载（读 agent.json，补全默认值）
├── session.ae   会话历史：增/查/渲染/持久化
├── ui.ae        终端界面：横幅、菜单、分隔线、角色标签
├── models.ae    可选模型清单
└── leash.json   注入全局变量 model / api_key / base_url
````

跑法：

````bash
# 交互式 TUI（在真实终端里）
./leash examples/agent/main.ae

# 非交互（如 leash-studio 的 Ctrl+R、或管道）自动进入演示模式
(echo 1; echo 你好; echo q) | ./leash examples/agent/main.ae
````

### 2.1 多文件怎么拼起来

入口 `main.ae` 顶部：

````leash
package file
package os
package ai
package time
import "leash.json"
import "colorx"
import "config.ae"
import "session.ae"
import "ui.ae"
import "models.ae"
````

要点：

1. **原生包用 `package` 声明**：`file`/`os`/`ai`/`time` 由编译器识别，声明后它们的裸名
   函数（`read`/`write`/`env`/`chat`/`now`/`is_tty`…）全局可用。
2. **兄弟文件用 `import "xxx.ae"`**：`config.ae` 等是纯 Leash 文件，按**入口文件所在目录**
   解析（`examples/agent/`）。函数名全局共享，所以 `main.ae` 里直接调 `cfg_load`、
   `sess_add`、`ui_banner` 即可，不用写前缀。
3. **`import "leash.json"` 注入全局变量**：把 JSON 里的 `model`/`api_key`/`base_url`
   变成程序里的全局变量，`chat()` 直接读它们。

> 一个常见误区：子文件里**不要**再写 `requires file` 之类的能力声明，也别在子函数里直接
> 调 `out`/`in`——宿主的 io 能力只在 `fn main` 内有效。正确做法是子函数**算结果、返回**，
> 由 `main` 负责打印（参见 `session.ae` 的 `sess_render` 只返回字符串，不自己 `out`）。

### 2.2 入口 main.ae 的主流程

````leash
fn main
  let cfg_path = "agent.json"
  let cfg = cfg_load(cfg_path)
  let hist_path = cfg["history_file"]
  let hist = sess_load(hist_path)

  out(ui_banner())
  out(ui_menu())

  // 非交互环境自动跑演示，让"运行"面板也能看到效果
  if not is_tty()
    run_demo(cfg, hist, cfg_path, hist_path)
    return

  while true
    ...
````

两件事值得说：

- **`is_tty()` 区分交互/非交互**：这是 `os` 包的原生函数，返回 stdin 是否是真实终端。
  在 leash-studio 里点"运行"、或用管道/重定向喂数据时，stdin 不是 TTY，`in()` 会立刻
  返回空串。与其让程序"秒退"让人困惑，不如在非交互时直接跑一段 `run_demo()` 示例对话——
  这也是为什么你在编辑器里 Ctrl+R 看到的是演示而非可键入界面。
- **`cfg`/`hist` 在调用前先加载**：`cfg_load` 来自 `config.ae`，`sess_load` 来自
  `session.ae`，跨文件按名调用，毫无障碍。

交互循环里每个菜单项都对应一个 helper：菜单 1 做单轮 `chat`、菜单 2 做多轮（把历史拼进
提示词）、菜单 3 渲染历史、菜单 4 切换模型、菜单 5/6 存/清历史。结构跟实例一的 `while +
else if` 一模一样，只是分支更多、逻辑拆到了别的文件。

### 2.3 配置：全局变量注入 + 运行时重赋值

`leash.json` 长这样（未填 api_key 时 `chat` 走 mock）：

````json
{
  "model": "gpt-4o-mini",
  "api_key": "",
  "base_url": ""
}
````

`import "leash.json"` 把它们变成全局变量。关键特性：**这些全局变量能在运行时重新赋值，
且 `chat()` 即时生效，无需重启**。看菜单 4：

````leash
model = nm                              // 直接改全局变量
write("leash.json", stringify({...}))   // 同时写回配置文件
cfg_save(cfg_path, {...})
out("已切换模型为 " + model + "（本次会话即时生效，并已写入配置文件）")
````

这条 `model = nm` 之所以能编译通过，是因为编译器对"来自 JSON 注入的全局变量"放开了赋值
（普通未声明全局默认不允许随意写，避免误改）。它背后接的是宿主的全局表，`chat()` 每次
调用都从那张表里读最新 `model`/`api_key`/`base_url`——所以演示模式里 `model = "claude-3-opus"`
之后，下一条 `chat` 立刻以 `claude-3-opus` 身份返回 mock 结果。

### 2.4 会话历史：把"记忆"拼进提示词

多轮对话的核心在 `session.ae` 的 `sess_context`：它把历史列表压成一段文本，塞回下一次
`chat` 的提示词里。菜单 2 的用法：

````leash
hist = sess_add(hist, "user", u)
let ctx = sess_context(hist, cfg["max_turns"])
let prompt = ctx + "用户: " + u
let reply = chat(cfg["system"], prompt)
hist = sess_add(hist, "assistant", reply)
````

`chat` 有两种签名：`chat(user)` 或 `chat(system, user)`。这里用第二种，把系统提示词
（`cfg["system"]`，来自 `agent.json`）和拼好的上下文一起传进去。没配 api_key 时，`chat`
返回 `[mock] 模型=... 收到问题: ...`，方便你在没有真模型的情况下把整个流程打通。

### 2.5 界面：颜色与分隔线

`ui.ae` 用 `colorx` 包（标准库，提供 `colorx_bold` 等）做高亮，用 `out("...")` 打印带
背景色的横幅。这些函数**返回字符串**，`main` 负责 `out`。这种"子模块只产数据/字符串、
入口负责 IO"的分层，正是多文件工程推荐的写法——既清晰，又因为 io 只在 `main` 里出现而
天然满足 Leash 的能力门控。

### 2.6 为什么文件读写要用相对路径

注意 `main.ae` 里运行时文件是 `agent.json` / `agent_history.json`（相对 cwd），而
`import "leash.json"` 是相对**入口目录**。两者在"从仓库根运行"和"从示例目录运行"时可能
指向不同位置。本项目约定：用 leash-studio 运行（cwd=示例目录）时文件落在示例目录内；
从仓库根运行时落在仓库根。这些运行时产物都已写入 `.gitignore`，不会污染提交。

## 3. 两个实例对照：从单文件到多文件

| 关注点 | 实例一 `todo.ae` | 实例二 `examples/agent` |
|--------|------------------|--------------------------|
| 文件数 | 1 | 6（入口 + 5 个模块） |
| 能力声明 | `package file` + `import "list"` | `package file/os/ai/time` + `import "colorx"` + 多 `.ae` |
| 数据 | list 装 map | list 装对话回合的 map |
| 持久化 | `write`/`read` + JSON | 同左，外加 `leash.json` 注入全局 |
| 交互 | `while true` + `in()` | `while true` + `is_tty()` 分流交互/演示 |
| IO 边界 | 全在 `fn main` | 子模块只算/只返回，`main` 统一打印 |

看出规律了吗？**多文件不是新语法，而是"把 `fn main` 里的 helper 挪到别的文件、用
`import` 连起来"**。先写通单文件，再按职责拆文件，是 Leash 工程化的标准路径。

1. **改实例一**：给 `todo.ae` 加一个 `r` 命令，删除第 n 项（提示：同样用"重建列表"的
   思路，跳过要删的那一项）。
2. **改实例一**：让任务支持优先级字段 `prio`，列表时按优先级排序（试试 `lib/list.ae`
   里的 `list_sort`）。
3. **读实例二**：在 `examples/agent/ui.ae` 里加一个彩色角色标签，让"助手"行显示成你
   喜欢的颜色。
4. **接真模型**：往 `examples/agent/leash.json` 填 `api_key` 和 `base_url`，把 `model`
   改成你可用的模型，再跑 `./leash examples/agent/main.ae`，看 `chat` 是否返回真实回答。

完成这些，你就已经从"会写语法"走到"能组织一个像样的 Leash 工程"了。下一步建议回到
《标准库详解》挑几个常用包（如 `string`/`list`/`dictx`）逐个试，把标准库变成肌肉记忆。
